# 售后排障手册

本文档面向实施、售后和维护人员，整理项目部署后最常见的问题与建议排查顺序。

## 1. 排障总原则

遇到问题时，建议先按这个顺序排查：

1. 先确认问题范围
2. 先看运维中心
3. 再看容器状态和日志
4. 再看配置和数据
5. 最后再考虑恢复或回滚

不要一上来就重装、删库或强行覆盖配置。

## 2. 基础排查入口

优先确认这些位置：

- 前台是否可打开
- `/api/health` 是否正常
- 后台是否可登录
- 运维中心是否可打开
- 操作日志是否可查看

## 3. 常见问题处理

### 3.1 后台登录失败

先检查：

- `.env` 中 `ADMIN_USERNAME`
- `.env` 中 `ADMIN_PASSWORD`
- 是否仍在使用旧 token
- 是否首次登录后已改密

如果表现为“之前能登，现在密码不对”：

- 先确认数据库里管理员密码状态是否已被修改
- 不要只盯着 `.env`，因为改密后实际以数据库状态为准

### 3.2 前台能开，后台打不开

优先检查：

- 前端路由是否正常加载
- 后端 `/api/admin/me` 是否返回 401 或 500
- 浏览器是否缓存旧 token

建议处理顺序：

1. 清理浏览器本地 token
2. 重新登录
3. 再看浏览器网络请求和后端日志

### 3.3 首页打不开或接口 502

优先检查：

```bash
docker compose ps
docker compose logs --tail=200 web
docker compose logs --tail=200 backend
docker compose logs --tail=200 mysql
```

重点看：

- `backend` 是否健康
- `web` 是否正确反向代理 `/api`
- `mysql` 是否完成启动

### 3.4 作品管理没数据

先分两类看：

1. 真的没有数据
2. 接口报错导致列表空白

建议排查：

- `/api/admin/flowers` 是否返回 200
- 当前是否筛到了 `deleted`
- 数据库里是否存在有效作品
- 后端是否有 SQL 或迁移错误

### 3.5 图片上传失败

优先检查：

- 上传目录是否存在
- 上传目录是否可写
- 磁盘空间是否足够
- 是否触发上传并发保护
- 图片大小是否超过 20MB

如果只有后台上传失败而前台正常：

- 重点检查管理员权限和登录状态

### 3.6 AI 生图不可用

先检查：

- AI 是否启用
- API Key 是否已配置
- 模型名是否正确
- base URL 和路径是否正确
- 运维中心里的 AI 状态是否正常

如果报密钥无效：

- 先确认后台保存的是最新 key
- 再确认模型和服务地址是否匹配

### 3.7 运维中心报错或无内容

优先检查：

- `/api/admin/system/status`
- `/api/admin/system/ops-tasks`
- 最近一次部署后是否遗漏环境变量
- 后端是否已完成启动

如果只有运维中心异常：

- 通常优先看系统状态接口和后端服务日志

### 3.8 操作日志无数据

先确认：

- 当前查看的是不是查询类动作
- 后台最近是否真的发生过写操作
- 是否已被归档

说明：

- 查询请求默认不记录日志
- 写操作、登录、恢复等动作才会记录

### 3.9 页面改了但线上没变化

先检查：

- 是否真的提交了代码
- 是否真的推送了远程
- 是否真的重新部署了容器
- 浏览器缓存是否仍是旧资源

建议排查：

1. 看 `git rev-parse --short HEAD`
2. 看后台运维中心里的提交号
3. 对比两边是否一致

### 3.10 新服务器部署失败

先检查：

- 当前用户是否有 `root` 或 `sudo`
- 是否可联网拉取基础镜像
- 是否能自动安装 Docker / Compose
- `.env` 是否缺关键字段

如脚本已自动安装依赖但仍失败：

- 重点看 Docker 服务是否已成功启动

## 4. 需要优先留意的高风险问题

以下问题优先级最高：

- 数据库连接异常
- 上传目录不可写
- 没有最近备份
- 默认管理员密码未修改
- JWT 密钥或数据加密密钥仍为默认值
- AI 已启用但密钥未配置

这些问题在运维中心通常都能直接看到风险提示。

## 5. 建议的恢复顺序

发生误操作时，建议优先顺序：

1. 先查操作日志
2. 能按日志恢复就先按日志恢复
3. 不行再考虑逻辑删除恢复
4. 仍不行再考虑配置导入
5. 最后才考虑整库备份恢复或版本回滚

这样影响范围最小。

## 6. 需要收集的售后信息

如果需要二线支持，至少先收集这些信息：

- 当前访问地址
- 当前部署时间
- 当前 git 提交号
- 问题出现时间
- 报错截图
- 浏览器网络请求报错
- 后端最近 200 行日志
- `docker compose ps` 输出

## 7. 建议常备命令

```bash
docker compose ps
docker compose logs --tail=200 web
docker compose logs --tail=200 backend
docker compose logs --tail=200 mysql
./backup.sh
./rollback.sh --latest --dry-run
./restore.sh --latest --dry-run
```

如果是 release 模式部署，还常用：

```bash
./release-status.sh
./release-inspect.sh
./release-rollback.sh --latest-previous
```
