# 花语时光作品库扩充设计

日期：2026-05-14

## 目标

将现有作品库补充到 72 个作品，覆盖 6 个实际业务分类，每个分类 12 个作品，并保证图片、文案、价格带、标签和精选策略都符合“花语时光”当前系统的品牌风格与展示定位。

本次工作以“前台展示完整、后台维护自然、整体视觉统一”为主，不引入新的业务分类，不改变现有作品数据结构，不直接修改数据库表结构。

## 范围

本次只处理以下内容：

- 补齐 6 个实际分类的作品数据，总量 72 个
- 为每个作品配置 2-3 张与分类匹配的图片
- 补齐名称、价格、描述、花材、寓意、标签、精选状态、排序和创建时间
- 通过现有后端接口写入数据
- 在当前 Docker 运行环境中验证前台画廊、详情页和后台作品管理表现

本次不处理：

- 新增分类
- 修改现有页面结构
- 修改数据库 schema
- 直接手改数据库记录
- 提交 `.env` 或 `flower-shop-backend-java/uploads/`

## 现状

当前系统已有分类如下：

- `wedding` 婚礼系列
- `daily` 日常花礼
- `opening` 开业花篮
- `seasonal` 节气花束
- `custom` 定制花艺
- `preserved` 永生花系列

现有作品数量明显不足，无法支撑完整画廊浏览、相关推荐、首页精选和后台统计的展示密度。当前前台与后台已经具备稳定的数据展示和作品 CRUD 入口，因此最稳妥的方式是沿用现有接口批量补齐作品。

## 数据设计

### 分类与数量

- 保持现有 6 个实际分类不变
- 每个分类补齐到 12 个作品
- 总计 72 个作品

### 作品字段

每个作品保持现有结构：

- `id`
- `name`
- `categoryId`
- `images`
- `price`
- `description`
- `materials`
- `meaning`
- `tags`
- `featured`
- `sort`
- `createdAt`

### 图片数量

- `婚礼系列 / 开业花篮 / 定制花艺`：每个作品 3 张图为主
- `日常花礼 / 节气花束 / 永生花系列`：每个作品 2 张图为主

### 命名与文案规则

- 名称长度控制在 4-8 个字
- 名称风格保持品牌化、花艺化，不采用电商促销标题
- 描述控制在 40-70 字，强调使用场景、花艺气质、色彩和氛围
- `materials` 保持 4-6 项
- `tags` 保持 3-4 项
- `meaning` 每个作品独立撰写，避免模板化重复

### 价格带规则

- `婚礼系列`：799 - 1699
- `日常花礼`：198 - 399
- `开业花篮`：598 - 1288
- `节气花束`：258 - 468
- `定制花艺`：999 - 2399
- `永生花系列`：299 - 699

### 精选与排序

- 每个分类 4 个精选作品
- 总计 24 个精选作品
- 排序区间按分类内梯度控制：
  - `100-89`：精选优先
  - `88-77`：普通核心款
  - `76-65`：延展款

### 创建时间分布

- `createdAt` 分散到 2026-01 至 2026-05
- 不集中在同一天，保证前后台都呈现“持续上新”的自然状态

## 图片策略

采用统一风格图片策略，而不是杂乱拼接多来源素材。

目标风格：

- 真实花艺摄影感
- 门店、桌面、自然光或柔和商业布光
- 色彩克制、构图整洁
- 符合“花语时光”的清新自然和品牌展示系统定位

图片来源执行策略：

1. 优先使用统一生成或统一筛选的同风格图片集
2. 统一落到当前 Linux 环境的项目目录中
3. 通过现有上传接口或当前可访问静态路径写入最终 URL
4. 不把素材分散挂在随机外链上，避免未来展示失效

## 实施方案

### 推荐方案

使用“批量生成统一风格图片 + 批量写入作品数据”的方案。

原因：

- 风格一致性最好
- 适合当前系统作为品牌展示站点
- 有利于首页、画廊、详情页的整体观感

### 执行步骤

1. 为 6 个分类定义统一图片风格约束和作品内容模板
2. 生成或整理图片素材，按分类与作品编号归档
3. 组装 72 个作品的数据 payload
4. 通过现有后端 API 批量创建作品
5. 在 Docker 环境中重建并验证前台与后台展示
6. 根据结果微调精选、排序和个别文案

## 错误处理

### 图片问题

- 若单张图片风格明显偏离分类主题，则剔除并替换
- 若图片 URL 不可访问，则重新上传并更新作品

### 数据写入问题

- 若批量创建过程中出现重复 ID，则按分类和序号重新生成唯一 ID
- 若接口校验失败，则修正字段长度、数组内容和必填项后重试

### 展示问题

- 若画廊页出现局部重复感过强，则重新分配创建时间、精选状态或图片顺序
- 若相关推荐相关性过弱，则通过标签与分类维持相关推荐质量

## 验证计划

完成实施后至少验证：

- `GET /api/flowers` 能返回扩充后的完整列表
- 前台 `/gallery` 分类筛选显示正常
- 前台 `/gallery/:id` 详情图、标签、相关推荐显示正常
- 首页精选区展示自然
- 后台 `/admin/flowers` 中作品统计、筛选、编辑入口正常
- Docker 环境下前端重新构建成功

## 风险与取舍

### 风险

- 大批量补作品会增加前台首屏后续滚动浏览密度，需要保证图片质量而不是只补数量
- 如果图片源风格不统一，会直接拉低整个系统的品牌感
- 如果一次性把全部作品都设为高权重，会导致首页和后台统计失真

### 取舍

- 优先保证风格统一，再追求速度
- 优先走现有 API，而不是直接改数据库
- 优先做完整可展示的 72 个作品，再做后续个别精品化微调

## 实施完成标准

满足以下条件才算完成：

- 6 个实际分类均达到 12 个作品
- 总作品数达到 72 个
- 图片可访问、风格统一、内容符合各自分类
- 前后台页面展示无结构性错误
- 构建验证通过
