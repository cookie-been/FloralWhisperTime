# 花语时光高并发防护与高可用预留设计

## 1. 背景与目标

当前系统默认部署形态为单机单实例 `Web + Java Backend + MySQL`。现有后端具备基础上传限制、AI 请求超时和 JWT 无状态认证，但尚未形成完整的高并发防护体系。在短时间内大量请求涌入时，系统存在以下风险：

- Web 线程被慢请求或高成本请求占满
- 数据库连接池被热点查询和高并发写操作耗尽
- AI 生图、上传、配置导入等高成本接口抢占全部资源
- 公开浏览接口被恶意或异常流量放大，拖慢后台和关键写接口
- 统一过载响应和运维可见性不足，排障只能依赖日志

本轮目标不是直接构建分布式高可用集群，而是先把单机单实例形态做到“可限流、可快速失败、可观察、可扩展”，同时为后续横向扩容保留清晰演进路径。

## 2. 设计边界

本轮确认的约束如下：

- 当前部署基线：单机单实例
- 目标：优先解决短时洪峰和资源耗尽问题，同时预留后续扩容
- 允许新增成熟依赖
- 保护策略：全站基础限流 + 管理后台更严格 + AI/上传等高成本接口最严格

本轮不做：

- Redis 分布式限流
- MQ 削峰
- 多机多实例编排
- 分布式缓存
- 复杂异步任务中心

## 3. 方案对比

### 方案 A：应用内分层保护 + 参数治理

在 Java 后端内部完成限流、并发隔离、容器参数治理、连接池治理、只读缓存和统一过载响应。

优点：

- 适合当前单机单实例形态
- 改动集中在应用层，实施成本低
- 可快速提升抗短时洪峰能力
- 后续可以平滑替换为网关/Redis 方案

缺点：

- 限流状态保存在本机内存，暂不适合多实例共享

### 方案 B：前置代理保护为主，应用兜底

在 Nginx 层处理主要限流和连接控制，应用层只保留少量兜底逻辑。

优点：

- 更接近未来多实例生产架构
- 入口保护能力更强

缺点：

- 当前单机单实例阶段收益不如方案 A 直接
- 需要同步改部署与反向代理配置

### 方案 C：直接上分布式保护栈

引入 Redis 分布式限流、异步任务化、队列削峰等。

优点：

- 扩展性最好

缺点：

- 当前明显超配
- 维护复杂度和交付成本偏高

### 推荐结论

本轮采用 **方案 A**，但所有保护阈值、分组规则和限流组件都以可替换、可扩展为前提设计，保证后续能够平滑演进到 `Nginx 前置限流 + 多实例` 或 `Redis 分布式限流`。

## 4. 总体设计

系统将增加 6 层保护：

### 4.1 路由分级限流

按请求路径将接口划分为 4 组：

1. `public-read`
   - `/api/health`
   - `/api/categories`
   - `/api/flowers`
   - `/api/flowers/**`
   - `/api/site-config`
   - `/api/shop-info`
   - `/api/brand-story`
   - `/api/about-page`
   - `/api/about-timeline`
   - `/api/team`

2. `public-write`
   - `/api/contact`

3. `admin`
   - `/api/admin/**`
   - 不含 AI、生图、上传、配置导入等高成本子路径

4. `heavy`
   - `/api/admin/ai/**`
   - `/api/uploads`
   - `/api/admin/system/config-import`

限流维度：

- 第一阶段按 `客户端 IP + 路由组` 进行本机限流
- 保留后续切换为 `用户标识 + IP + 路由组` 或 Redis key 的能力

超限策略：

- 统一返回 `429`
- 响应体仍沿用 `{ "message": "..." }`

### 4.2 高成本接口并发隔离

对高资源消耗接口增加独立并发上限，不允许它们占满 Tomcat 工作线程和数据库连接池：

- AI 生图：单独信号量保护
- 图片上传：单独信号量保护
- 配置导入：串行或极低并发

并发满时：

- 不等待长队列
- 直接快速失败
- 返回 `503` 或明确业务提示

### 4.3 容器与连接池参数治理

在 `application.yml` 中引入可配置的 Tomcat 和 Hikari 参数，所有阈值支持环境变量覆盖。

目的：

- 限制 Web 线程无界膨胀
- 控制请求堆积深度
- 确保数据库连接池在高压下可预测
- 通过短超时让系统更快恢复

### 4.4 只读热点缓存

对高频、低变、后台修改后可主动失效的公开读接口使用本地缓存减压：

- 站点配置
- 门店信息
- 品牌故事
- About 页
- 时间轴
- 团队成员
- 分类

缓存要求：

- Caffeine 本地缓存
- TTL 短，保证前后台一致性可接受
- 后台更新后主动失效

### 4.5 统一过载响应与状态可见性

需要新增统一过载异常处理与状态输出：

- 限流触发：`429`
- 并发隔离/资源繁忙：`503`

同时在后台系统状态页增加保护能力摘要，例如：

- 是否启用基础限流
- 当前各路由组阈值
- 当前高成本接口并发上限
- 最近拒绝次数

### 4.6 后续扩容预留

本轮所有保护逻辑必须遵循以下约束：

- 不依赖本地 Session
- 保护阈值全部配置化
- 限流 key 构造器独立
- 路由分组规则独立
- 本机限流器实现与业务逻辑解耦

这样后续可演进为：

`单机内存限流`  
-> `Nginx 前置限流 + 应用兜底`  
-> `Redis 分布式限流 + 多实例`

## 5. 阈值建议

### 5.1 基础限流

默认建议值：

- `public-read`：每 IP `60 req / 10s`
- `public-write`：每 IP `12 req / 60s`
- `admin`：每 IP `30 req / 60s`
- `heavy`：每 IP `6 req / 60s`

说明：

- 当前值按“保守但能明显挡住瞬时洪峰”设定
- 上线后可根据真实流量再调高

### 5.2 并发隔离

默认建议值：

- AI 生图同时执行数：`2`
- 上传同时执行数：`4`
- 配置导入同时执行数：`1`

### 5.3 Servlet 容器

Tomcat 建议初始值：

- `server.tomcat.threads.max = 80`
- `server.tomcat.accept-count = 200`
- `server.tomcat.max-connections = 1000`
- `server.tomcat.connection-timeout = 5s`

### 5.4 数据库连接池

Hikari 建议初始值：

- `maximum-pool-size = 20`
- `minimum-idle = 5`
- `connection-timeout = 3000ms`
- `validation-timeout = 1000ms`
- `idle-timeout = 600000ms`
- `max-lifetime = 1800000ms`

## 6. 模块设计

### 6.1 限流模块

新增独立限流组件，职责包括：

- 路由组识别
- 客户端标识提取
- 令牌桶获取与消费
- 统计拒绝次数

建议以 Spring MVC `HandlerInterceptor` 或 `OncePerRequestFilter` 落地。优先使用拦截器，便于按业务路径做清晰分流。

### 6.2 并发隔离模块

新增独立保护组件，对 AI、生图、上传、配置导入这类接口提供：

- 资源名
- 并发阈值
- `tryAcquire` / `release`
- 拒绝计数

控制器只负责调用保护器，不承担并发逻辑。

### 6.3 缓存模块

在 `SiteService` 相关只读接口前增加统一缓存服务或局部缓存包装，避免将缓存逻辑散落到多个 Controller。

要求：

- key 简单明确
- 后台写操作后集中失效
- 不缓存敏感后台接口

### 6.4 状态输出模块

扩展现有系统状态响应 DTO，新增保护相关状态字段，用于后台系统页展示和运维验收。

### 6.5 统一异常响应

在现有全局异常处理中新增：

- 限流异常
- 并发隔离异常

两者都保持前端兼容的 `{ "message": "..." }` 返回格式。

## 8. 错误处理策略

### 8.1 限流

返回：

```json
{ "message": "当前请求较多，请稍后重试" }
```

HTTP 状态码：`429`

### 8.2 高成本接口拥塞

返回：

```json
{ "message": "当前生成任务较多，请稍后再试" }
```

或：

```json
{ "message": "当前服务繁忙，请稍后重试" }
```

HTTP 状态码：`503`

### 8.3 降级原则

- 不做隐式吞错
- 不把高压问题伪装成普通业务失败
- 给前端明确、可提示的过载信息

## 9. 测试与验收

### 9.1 后端测试

新增测试覆盖：

- 不同路由组限流命中
- 超限返回 `429`
- 高成本接口并发隔离返回 `503`
- 缓存命中与失效
- 系统状态输出新增保护字段

### 9.2 构建验证

- 后端：`mvn test`
- 前端：`npm run build`

### 9.3 人工压测验收

建议通过简单压测工具验证：

- 公开读接口被快速突发访问时服务不崩
- AI 接口并发满时只拒绝新增请求，不拖垮其他接口
- 后台系统状态页可看到保护策略和拒绝计数

## 10. 演进路线

后续扩展顺序建议：

1. 单机应用内保护稳定运行
2. Nginx 增加前置连接控制和限流
3. Docker Compose 双实例 + 负载均衡
4. 将限流状态从本机内存切换到 Redis
5. 针对 AI 等高耗时任务逐步引入异步任务化

## 11. 结论

本设计以“单机单实例先稳住”为核心，不追求一步到位的分布式复杂方案，而是优先补齐最关键的：

- 入口限流
- 高成本接口并发隔离
- 容器与连接池上限治理
- 热点只读缓存
- 统一过载响应
- 可观察、可扩展的状态输出

这样既能显著降低短时洪峰导致服务挂机的风险，也能为后续高可用和水平扩展留下清晰、低返工的演进路径。
