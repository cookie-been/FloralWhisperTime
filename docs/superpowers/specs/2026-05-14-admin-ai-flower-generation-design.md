# 后台 AI 作品生成设计文档

日期：2026-05-14

## 1. 目标

在后台 `作品管理` 中新增一个 AI 生成入口，支持管理员通过自由文本 prompt 发起文生图，或上传最多 3 张参考图发起以图生图。生成结果不直接进入正式作品库，而是先作为可预览结果展示给管理员，管理员确认后再进入现有“新增作品”流程并发布。

本期目标是把 AI 生成结果稳定接入当前作品管理工作流，而不是做一个独立的 AI 创作系统。

## 2. 范围

### 2.1 本期包含

- 后台 `AdminFlowers` 页面新增 `AI生成作品` 入口
- 独立 AI 生成对话框
- 支持自由输入 prompt
- 支持上传最多 3 张参考图
- 单张参考图大小限制 20MB
- 支持文生图和以图生图
- 每次只生成 1 张结果图
- 后端按火山引擎官方教程接入 `Doubao-Seedream-5.0-lite`
- 生成后的图片下载并保存到本地 `uploads/ai/`
- 生成结果预览
- 点击“用于新建作品”后，自动带入现有新增作品表单

### 2.2 本期不包含

- 批量生成
- 多轮会话式 AI 对话
- AI 草稿持久化表
- AI 生成历史查询
- AI 自动生成完整作品文案
- AI 直接自动发布作品
- 多模型切换

## 3. 用户流程

### 3.1 文生图

1. 管理员进入后台作品管理页
2. 点击 `AI生成作品`
3. 在对话框中输入 prompt
4. 不上传参考图
5. 点击生成
6. 系统调用豆包模型生成图片
7. 后端下载图片到本地 `uploads/ai/`
8. 前端展示生成结果预览
9. 管理员点击 `用于新建作品`
10. 系统打开现有新增作品抽屉，并把生成图写入 `images`
11. 管理员继续填写名称、分类、价格、描述等字段
12. 管理员保存后，作品进入正式作品库

### 3.2 以图生图

1. 管理员进入后台作品管理页
2. 点击 `AI生成作品`
3. 输入 prompt
4. 上传 1-3 张参考图
5. 点击生成
6. 系统调用豆包模型生成图片
7. 后端下载图片到本地 `uploads/ai/`
8. 前端展示生成结果预览
9. 管理员点击 `重新生成` 或 `用于新建作品`

## 4. 交互设计

### 4.1 后台入口

在 `AdminFlowers` 页面工具栏增加一个新按钮：

- 按钮文案：`AI生成作品`
- 位置：与现有新增作品按钮同层级

### 4.2 AI 生成对话框

对话框包含：

- `prompt` 多行输入框
- 参考图上传区
- 参考图数量提示
- 参考图大小限制提示
- 生成按钮
- 生成中状态
- 错误提示区
- 结果预览区
- `重新生成` 按钮
- `用于新建作品` 按钮

### 4.3 审核与发布

本期不引入独立“AI 草稿”概念。

生成结果只存在于当前前端会话中：

- 管理员认可结果时，点击 `用于新建作品`
- 系统将生成图注入现有新增作品表单
- 管理员按现有流程完成正式保存

这样可以最大限度复用当前作品管理逻辑，避免多引入一套草稿生命周期。

## 5. 后端设计

### 5.1 模块划分

建议新增三个后端责任单元：

1. `AI 配置层`
   - 读取火山引擎密钥、模型名、上传限制等配置

2. `火山引擎 Provider 层`
   - 负责构造官方 API 请求
   - 处理文生图 / 以图生图差异
   - 解析响应

3. `AI 生成管理接口`
   - 供后台调用
   - 接收 prompt 和参考图
   - 调用 provider
   - 下载图片到本地
   - 返回本地图片 URL

### 5.2 新增接口

建议新增：

#### `POST /api/admin/ai/images/generate`

请求类型：

- `multipart/form-data`

请求字段：

- `prompt`: string，必填
- `referenceFiles`: file[]，可选，最多 3 张

返回字段建议：

```json
{
  "success": true,
  "imageUrl": "/uploads/ai/xxx.png",
  "source": "doubao-seedream-5.0-lite",
  "mode": "text_to_image"
}
```

或：

```json
{
  "success": true,
  "imageUrl": "/uploads/ai/xxx.png",
  "source": "doubao-seedream-5.0-lite",
  "mode": "image_to_image"
}
```

## 6. 豆包接入策略

### 6.1 模型

- 模型名：`Doubao-Seedream-5.0-lite`

### 6.2 接入原则

- 严格按火山引擎官方教程和 API 参考实现
- 不自行发明协议
- 不在 controller 中直接拼接三方请求
- 三方调用逻辑单独封装

### 6.3 密钥处理

密钥只通过环境变量注入，例如：

- `VOLCENGINE_API_KEY`
- `VOLCENGINE_IMAGE_MODEL`

禁止：

- 写入代码
- 写入前端
- 写入仓库
- 打印完整密钥到日志

### 6.4 图片落地

无论豆包返回的是可下载 URL 还是其他可转换格式，最终系统内部都统一做成：

- 后端下载/转换图片
- 存储到 `flower-shop-backend-java/uploads/ai/`
- 返回站内可访问 URL

这样可以保证：

- 前台和后台图片访问方式统一
- 避免长期依赖第三方外链
- 后续更容易切换到 OSS / CDN

## 7. 文件与存储设计

### 7.1 参考图

参考图仅用于本次生成调用，不进入正式作品库。

本期不要求对参考图做长期持久化管理；是否需要临时存储，取决于火山引擎接口调用方式：

- 如果后端可直接转发 multipart 文件，则无需单独落盘
- 如果调用链要求先转为可读取文件流，则只保留请求过程中的临时文件

### 7.2 生成图

生成图永久保存到：

```text
flower-shop-backend-java/uploads/ai/
```

命名建议：

- `ai-时间戳-随机串.png`

## 8. 与现有系统的衔接

### 8.1 作品管理页

现有 `AdminFlowers` 已有新增作品抽屉，本期不新建第二套作品发布表单。

AI 生成功能只负责提供：

- 生成图片
- 把图片注入新增表单

正式作品数据仍通过现有：

- `createFlower`
- `updateFlower`

流程写入。

### 8.2 数据结构

本期不修改正式作品表结构。

AI 生成结果进入正式流程时，只需要为现有 `Flower` 结构填充：

- `images`

其余字段继续由管理员手工填写。

## 9. 校验与错误处理

### 9.1 前端校验

- prompt 不能为空
- 参考图数量不能超过 3
- 单张大小不能超过 20MB
- 上传格式限制为常见图片格式

### 9.2 后端校验

- 二次校验 prompt 非空
- 二次校验参考图数量
- 二次校验单张大小
- 二次校验 MIME 类型

### 9.3 第三方调用失败

需要处理：

- 鉴权失败
- 请求超时
- 模型限流
- 响应格式异常
- 图片下载失败

返回给前端的错误仍保持当前系统统一格式：

```json
{ "message": "..." }
```

### 9.4 重试体验

如果生成失败：

- 保留当前 prompt
- 保留当前参考图列表
- 管理员可直接再次点击生成

## 10. 安全与成本控制

### 10.1 安全

- 仅管理员可调用生成接口
- 密钥只存在后端
- 不暴露第三方真实调用参数给前端

### 10.2 成本控制

本期最小控制策略：

- 单次只生成 1 张
- 最多 3 张参考图
- 单张最大 20MB
- 后续可增加频率限制与调用日志

## 11. 测试与验收

### 11.1 后端验收

- 纯文本生成成功
- 带 1 张参考图生成成功
- 带 3 张参考图生成成功
- 超过数量限制时报错
- 超过 20MB 报错
- 豆包失败时前端收到明确错误
- 生成图成功落到 `uploads/ai/`

### 11.2 前端验收

- 对话框可打开关闭
- 上传列表正确显示
- 生成中状态明确
- 结果图可预览
- 点击“用于新建作品”后，新增抽屉能自动带入图片

### 11.3 Docker 验收

- 容器内可访问火山引擎接口
- 生成结果能通过 Web `/uploads/...` 访问
- 不影响现有作品管理、上传、前台浏览链路

## 12. 后续演进

如果第一期稳定，可在后续扩展：

- AI 生成草稿表
- 生成历史记录
- 多候选图
- AI 辅助生成标题和描述
- 批量生成
- 云存储替代本地 uploads

## 13. 推荐结论

推荐采用“轻量接入 + 人工审核发布”的方案：

- 风险低
- 改动面可控
- 与现有后台工作流贴合
- 最适合当前系统阶段
